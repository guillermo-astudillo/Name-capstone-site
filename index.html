Concept

A privacy project about recognition, context, and agency.

NAME frames digital identity as both a privacy issue and an identity-psychology issue. The workflow supports users in identifying what is attached to their name, deciding what should remain associated with it, and moving through one realistic action cycle at a time.

Digital identity as a privacy issue

Recognition increasingly begins with search results rather than direct interaction. NAME treats that shift as actionable design terrain.

Context matters

Information created for one purpose can become the lens for another. The product helps restore fit between context and exposure.

Procedural realism

The system organizes existing rights and procedures rather than promising perfect erasure or impossible control.

Design with boundaries

Visible limits, user confirmation, proof-of-control checks, and logs make safety part of the product itself.

Workflow

The report, rebuilt as a website.

Each section below adapts one page of the capstone into a clean, web-native block.

01

Overview

This opening screen introduces NAME as a guided workflow for managing name-based digital identity risk.

Digital identity is treated here as both a privacy issue and an identity-psychology issue.
Guided framing for a manageable first step
Map, Decide, and Track as the core workflow
One end-to-end action cycle instead of abstract privacy talk
A clearly user-facing interface centered on agency
02

Onboarding & Scenario Triage

The workflow begins by reducing uncertainty through issue classification before any action is suggested.

Behavioral design lowers ambiguity so users can move from uncertainty into a concrete first action.
Scenario-based start lowers cognitive load
A single next step creates a low-friction first win
Reduced visibility is distinguished from total deletion
The app becomes intelligible before technical action begins
03

Identity Map

The Identity Map inventories the traces that shape a searchable public identity across platforms, institutions, and aggregators.

Identity, impression management, and context-relative information flows are made operational through inventory and review.
Trace counts make scope visible at a glance
Each record is organized by source, confidence, and priority
Uncertain matches prevent false certainty
Human review is built in before action is generated
04

Decision Layer

Users determine whether a trace should be kept, limited, or severed from their searchable identity.

The keep-limit-sever structure is an applied model of context restoration and public identity governance.
Selected trace context anchors judgment
Keep, Limit, and Sever create a practical decision model
Priority and outcome connect choice to realistic next steps
The logic restores context rather than equating privacy with secrecy
05

Action Pathways

The system reorganizes existing rights, platform procedures, and privacy actions into clear procedural pathways.

The main design move is bundling rights and procedures into a legible interface with lower friction.
Removal request
Correction packet
De-indexing
Preventive settings
06

Worked Example

A concrete example shows how one trace moves through the workflow from discovery to documented action.

Worked examples improve classification, confidence, and actual use by showing one realistic path from discovery to action.
Five-step mini-flow
Clear case framing
Evidence requirements tied to stronger remedies
Status language keeps the process procedural and realistic
07

Action Tracker

The tracker helps users monitor requests, timelines, evidence, and escalation notes across multiple actions.

Auditability turns privacy repair into a process users can revisit, continue, and escalate over time.
Active item counts and progress snapshot
Structured rows with trace, action, and status
Waiting periods become visible instead of vague
Evidence and escalation notes stay in one place
08

Action Plan Export

The export feature turns the workflow into a portable one-page summary that can move across contexts.

Documentation allows the workflow to travel across advising, legal, and institutional settings without losing continuity.
Goal statement
Top traces this week
Recent actions already completed
Next steps and follow-up schedule
09

Visible Limits & Safety

The system openly states what it can and cannot do and builds user protections directly into the workflow.

Safety by design means being honest about legal and technical limits while resisting abuse and overclaiming.
Visible exclusions up front
User confirmation for uncertain traces
Proof-of-control checks for high-risk actions
Tamper-evident logs and escalation instead of magic promises

For users

A clearer route through search results, old profiles, directories, and misidentification problems.

For advisors and clinics

A portable action record that supports follow-up rather than one-off conversations.

For institutions

A stronger model for translating privacy, documentation, and procedural support into service design.
